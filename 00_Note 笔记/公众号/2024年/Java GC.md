失踪人口回归

忍不住给大家分享一个JVM调优案例，这是我第n次通过调试JVM参数以解决业务效率问题。过程坎坷，最终结局是好的。

#### 项目背景-报表系统

背景是一个报表系统，这是一个整合企业所有业务的综合报表系统，我们作为使用方，需要每年、每台（每个环境都要付费）给报表研发公司不少的版权及运维费用。应用服务是部署在公司内部服务器的，因为没有源码，一般出现我们无法修复的问题，需要提交工单给研发公司（这一部分额外收费）。

报表的主要功能是通过配置不同数据源（Oracle、MySQL、ClickHouse等多种数据库），然后通过数据开发人员连接数据源，开发复杂查询条件为企业提供战略决策。这一切操作都在系统中进行，因此对系统的可靠性、可用性十分依赖。随着时间不断推迟，业务不断扩展，报表数量越来越多，查询条件越来越复杂，系统逐渐变得繁重。

#### 初步问题发现

2024年5月左右，企业告警群突然收的生产环境系统宕机信息，浏览器前端访问系统后，发现可以正常打开，也可以正常操作，只是不知道为什么会触发宕机告警。

一开始我们只猜测是因为 Prometheus 监控系统的黑盒在探测应用是否存活时，出现秒级的网络中断，因为监控系统和应用是两个不同的网段，分别是 192.168.1.X/24 和 192.168.2.X/24 。所以当时的结论就怪在网络上了。

> 技术不行，就怪服务器、网络，最后就怪业务数据

给出的方案是调整 Prometheus 重试时间，从默认10s，修改到30s。30秒内发现服务在线都会判定为非宕机状态。

告警确实消停了一阵子，时间来到6月，告警依旧持续，以每天 1、2次的频率发着。

但告警发出后，系统依旧可以访问，没有发现任何不对劲的地方。

#### 重视问题

2024年9月，业务人员反馈系统无法登录，我们终于重视起来了。为什么我们用的时候好好的，业务用的时候就用不了呢，此时还没有往 STW 方向考虑，领导提交了个工单，让我对接系统的研发公司，讨论问题如何修复。

其中我还做了如下多种不同测试网络连接性测试

1. 通过 crontab 定时器，每秒 crul 系统，将返回的code输出到服务器内日志，每分钟1次。现象就是所有请求都是正常的。
2. 通过多个不同网段服务器对步骤1进行测试
3. 排查应用系统的CPU、内存情况

终于，在排查内存情况时，使用 jstat命令发现应用FCGT（Full GC Time） 较大

```sh
jstat -gcutil $pid 1000 3
# 每隔1000毫秒打印1次GC情况，共打印3次
```

终于将问题定位到GC情况，查看改应用历史配置，发现设置为最大最小内存为 40G
